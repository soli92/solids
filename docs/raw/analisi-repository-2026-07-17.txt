## Parte 2 — Concetti e conoscenza estratta

### 2.1 Architettura tema

SoliDS usa l'attributo HTML `data-theme` per il switching dei temi. L'approccio è:
1. Definizione token base (valori assoluti, es. `color.gold.500 = #C9A227`)
2. Definizione token semantici per ogni tema (es. `color.text.primary = {gold.500}` per tema fantasy)
3. Generazione CSS variables `--sd-*` via build pipeline (Style Dictionary implicita)
4. I consumer usano solo le CSS variables semantiche — mai valori hardcoded

Temi supportati (12):
- Light, Dark (default Material Design 3 oriented)
- Fantasy, Cyberpunk, 90s-Party, Steampunk (temi creativi)
- Captain America, Ichigo, Inuyasha, Sasuke, Vegeta, Zoro (temi personaggio)

### 2.2 Sistema icone (post-bonifica)

- `src/icons/glyphs.tsx` — icone base universali (Home, Search, User, ecc.) — ~246 righe
- `src/icons/glyphs-themed.tsx` — icone tematiche (Fantasy×3, Cyberpunk×3, Party×3) — source of truth post-bonifica
- `src/icons/solids-icon.tsx` — wrapper `SolidsIcon` con token CSS
- `src/icons/index.ts` — re-export di tutti e tre

**Bridge registry:**
- `src/lib/icons/index.ts` — re-export da `@/icons` — bridge per compatibilità path consumer registry

**Export npm:**
- `@soli92/solids/icons` — tutte le icone (SolidsIcon + base + tematiche + brand)

### 2.3 Registry shadcn (Model 1)

Il registry segue il Model 1: un kit `solids-ui` che aggrega tutti i componenti più una dipendenza `solids-icons` per le icone tematiche. File chiave:

- `scripts/sync-registry.mjs` — script che rigenera i JSON — modifiche manuali a `registry.json` vengono sovrascritte ad ogni run

**Il problema principale è `.cursor/rules/wiki.md` con `alwaysApply: true`.**

Questo file inietta in ogni sessione Cursor la regola:
> "All behavioral rules for this repository live in AGENTS.md. Read AGENTS.md before doing anything."

Il puntamento è a `docs/wiki/AGENTS.md` (schema Karpathy), non a `AGENTS.md` root (factory).
Il `docs/wiki/AGENTS.md` definisce:
- Wiki root: `docs/wiki/` (conflitto con `wiki/` factory)
- Raw folder: `docs/raw/` (coincide per caso, ma con schema diverso)
- Struttura: sources/, concepts/, decisions/, runbooks/, incidents/, components/, syntheses/ dentro `docs/wiki/`
- Regola agent: scrivere SOLO dentro `docs/wiki/` — conflitto diretto con factory

**Il `docs/wiki/AGENTS.md` è incompatibile con il pattern factory** su questi punti:
1. Definisce un'architettura a 3 layer diversa dalla factory
2. Impone che il wiki sia dentro `docs/wiki/` invece di `wiki/` root
3. Usa linking in stile Obsidian (`[[wikilinks]]`) vs frontmatter factory
4. Non conosce il concetto di wiki-keeper, wiki-query, wiki-lint
5. Ha un loop operazionale diverso (ingest/query/lint come comandi espliciti)

**Artefatto fantasma**: `docs/wiki/wiki/` è una wiki dentro docs/wiki/ — evidenza che qualcuno ha eseguito un ingest parziale seguendo lo schema Karpathy prima di adottare la factory, producendo un'anomalia strutturale.

**Azione raccomandata**: eliminare integralmente `docs/wiki/`.

## Appendice — Evidenza docs/wiki/ incompatibilità

### Confronto schema Karpathy (docs/wiki/AGENTS.md) vs factory (wiki-keeper)

| Dimensione | Schema Karpathy (ghost) | Factory pattern v2.33 |
|------------|------------------------|----------------------|
| Wiki root | `docs/wiki/` | `wiki/` (root) |
| Raw folder | `docs/raw/` | `docs/raw/` |
| Linking | `[[wikilinks]]` Obsidian | Path relativi markdown |
| Frontmatter | `type/created/updated/sources/status/language` | `id/type/title/status/created/updated/sources` |
| Agenti | Nessun agente dedicato | `wiki-keeper`, `wiki-query`, `wiki-lint` |
| Ingest | Comando manuale `Ingest docs/raw/<file>` | Slash command `/sync-docs` + wiki-keeper agent |
| Lint | Comando manuale `Lint the wiki` | `/lint` → wiki-lint agent |
| Query | Prompt esplicito | `/query` → wiki-query agent |
| Log | `docs/wiki/log.md` | `wiki/log.md` |
| Sottocartelle | sources/, concepts/, decisions/, runbooks/, incidents/, components/, syntheses/ | concepts/, entities/, runbooks/, decisions/, incidents/ |

### Attivazione Cursor rule (prova interferenza)

Il file `docs/wiki/.cursor/rules/wiki.md` contiene:
```yaml
---
description: LLM Wiki schema for this repository
alwaysApply: true
---
All behavioral rules for this repository live in AGENTS.md at the repo root.
Read AGENTS.md before doing anything else in this repo.
```

Con `alwaysApply: true`, Cursor inietta questa regola in ogni conversazione.
"AGENTS.md" in questo contesto punta a `docs/wiki/AGENTS.md` (l'unico AGENTS.md
visibile quando si naviga da `docs/wiki/`), che definisce un'architettura incompatibile.
Questo può causare che il modello scriva nella wiki sbagliata (`docs/wiki/`) invece di `wiki/`.
