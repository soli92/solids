Comando della capability [[accessibility-testing-capability]] (EP-007, terza forma del pattern
Tool / Skill / Agente). Delega lo scan a11y end-to-end (adapter detect → `run_a11y_scan` →
severity+WCAG → manual checks → report standard) all'agente [a11y-specialist](mdc:.cursor/rules/a11y-specialist.mdc); con fallback
sulla skill [accessibility-testing-protocol](mdc:.cursor/skills/accessibility-testing-protocol/SKILL.md) quando l'agente non è scaffoldato. Thin dispatcher,
analogo a `/review` (CQRL) e `/analytics`.

Riferimenti architetturali: ADR-014 (3 modalità d'uso, no owner unico, fallback discovery
deterministico), ADR-015 (fallback mobile/non-web inline), ADR-016 (config block `a11y:`,
side-channel storage, frontmatter TSK `a11y_status:`). PATTERN §3 (operazioni opzionali),
R.P3 (opt-in totale).

## Sintassi

```
/a11y <target> [--standard=wcag22aa] [--include-interactive] [--ephemeral]
```

## `<target>` — 4 forme

| Forma | Esempio | Significato |
|---|---|---|
| **URL http/https** | `/a11y https://app.example.com/checkout` | route/sito live (Playwright+axe diretto) |
| **percorso file** | `/a11y src/components/Button.tsx` | file componente isolato (harness o review statica) |
| **dir di build** | `/a11y dist/` | output statico/SSR (serve dir → URL → axe) |
| **TSK-id** | `/a11y TSK-039` | recupera `code_path` / route dal frontmatter del TSK |

Con **TSK-id**: `Glob management/kanban/**/TSK-<id>.md`, leggi il frontmatter e deriva il target
dai `code_path` (o dalla route associata). Se il TSK non esiste → ABORT «TSK non trovato».

## Flag

- `--standard=wcag22aa` — default `wcag22aa` (tag axe-core, passato al tool `run_a11y_scan`).
- `--include-interactive` — default `false`. Se presente abilita i check tastiera/focus/reflow
  (richiede browser reale via Playwright).
- `--ephemeral` — **non scrive** in `code_quality/reports/` (analogo a `/query --ephemeral`):
  il report è restituito solo in chat, non nel side-channel. L'entry su `wiki/log.md` viene
  comunque appesa.

## Comportamento

1. Read `factory.config.yaml.a11y`. Se `enabled: false` (o blocco assente) → ABORT pulito
   (capability spenta); suggerisci «Abilita con `a11y.enabled: true`».
2. Risolvi `<target>` nelle 4 forme sopra (se `TSK-id`, deriva il target dal frontmatter del TSK).
3. **Invocazione** (con fallback discovery, ADR-014, precedence ordinata):
   - Se [a11y-specialist](mdc:.cursor/rules/a11y-specialist.mdc) è scaffoldato **e** `a11y.agent: true` → invoca
     l'agente `a11y-specialist` passando `target`, `standard`, `include_interactive`, `ephemeral`.
   - **Fallback** (agente non scaffoldato / `agent: false`): se la skill
     [accessibility-testing-protocol](mdc:.cursor/skills/accessibility-testing-protocol/SKILL.md) (US-024) è presente → la skill è invocata da fe-dev o
     qa-dev (chi è attivo nella topologia). La capability resta disponibile come skill anche
     senza agente dedicato (PATTERN §3, albero Tool/Skill/Agente). Precedence: qa-dev su TSK FE
     done → fe-dev via skill US-024.
   - Se né agente né skill esistono → fail-loud «Nessun agente disponibile per a11y scan;
     topologia non compatibile. Vedi `factory.config.yaml.topology` e `a11y.agent`.»
4. L'agente/skill produce il report nello schema standard US-025
   (`{target, standard, summary, automated_findings, manual_checks, positive_findings}`),
   rispettando la **regola di neutralità** [[wcag-automated-coverage-limit]] (ADR-016 §G): mai
   dichiarare conformità sulla sola base degli automated findings; `manual_checks` mai vuota
   (N ≥ 1); formula obbligatoria *"Nessun errore automatico rilevato; restano N verifiche
   manuali"*.
5. Mostra in chat il summary (critical/major/minor + N manual_checks). Su TSK-id, l'agente può
   aggiornare il solo campo frontmatter `a11y_status:` del TSK target (mai il corpo).

## Storage

- **Con TSK-id** → `code_quality/reports/<TSK-id>-a11y-iter-<N>.{json,md}` (riuso path CQRL,
  ADR-016 §E; lo slug `a11y` distingue dagli iter `visual` e CQRL).
- **Standalone / ad-hoc** (URL, file, dir) → `code_quality/reports/_adhoc/a11y-<YYYY-MM-DD-HH-MM>-<slug>.{json,md}`.
- **`--ephemeral`** → **nessuna scrittura** in `code_quality/reports/` (solo chat + log).

## Logging

Ogni invocazione appende a `wiki/log.md` (anche con `--ephemeral`) una entry nel formato
canonico, analogo alle entry `review`:

```
[YYYY-MM-DD HH:MM] a11y <target> → <verdict>
```

## Fallback (agente non scaffoldato)

Se l'agente `a11y-specialist` **non** è scaffoldato ma la skill
[accessibility-testing-protocol](mdc:.cursor/skills/accessibility-testing-protocol/SKILL.md) (US-024) **sì**, il comando non fallisce: la skill è invocata
da fe-dev o qa-dev attivi nella topologia (ADR-014 §Fallback discovery). La forma Agente è
una comodità di delega, non un prerequisito della capability.

## Vincoli (R.P3 — opt-in totale)

- Comando **opt-in**: la sua assenza non è ERROR di lint (R.P3). Assenza di **entrambi** i file
  (agente + comando) → comportamento orchestrator identico a v2.17 (backward compat, 0 nuove
  ERROR/WARNING).
- **Mai** dichiarare conformità WCAG / legale: la capability è pre-screening interno (regola di
  neutralità, enforced dall'agente/skill).
- L'agente non modifica il corpo dei TSK: al massimo il campo frontmatter `a11y_status:` del TSK
  target (single-writer logico, ADR-014/016).

`tools/a11y/a11y-scan.sh`, EP-007, ADR-014/015/016, e [[accessibility-testing-capability]]
per il contratto completo.
